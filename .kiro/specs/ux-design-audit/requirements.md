# Requirements Document

## Introduction

Системный UX/дизайн-аудит сайта AI Каталог СНГ на основе философии и компонентов Intuition Design System (IDS). Цель — выявить абсурдные дизайн-решения, перегруз UX, неконсистентность дизайна и привести сайт к единой системе.

## Glossary

- **IDS (Intuition Design System)**: Дизайн-система для вёрстки лендингов и онлайн-изданий, основанная на нарративах. Использует функциональное именование цветов, систему "этажей" вместо модульных сеток, и компоненты: Wrapper, Space, TextWidth, Sequence, Rounded.
- **Density**: Базовая переменная IDS (`--ids__density: 1.3`) для расчёта всех отступов и размеров.
- **Wrapper**: Контейнер с фиксированной шириной (70% на десктопе) для организации "этажей" страницы.
- **TextWidth**: Ограничитель ширины текста (85% от Wrapper) для комфортного чтения.
- **Space**: Вертикальные отступы с размерами S/M/L/XL, рассчитанные через density.
- **Sequence**: Flex-контейнер для горизонтальных последовательностей элементов.
- **Rounded**: Компонент для скругления углов суперэллипсом (squircle).
- **Surface**: Цвет фона для карточек и блоков (`--ids__surface-RGB`).

## Requirements

### Requirement 1: Неконсистентность заголовков

**User Story:** Как пользователь, я хочу видеть единообразную типографику заголовков, чтобы легко ориентироваться в иерархии контента.

#### Acceptance Criteria

1. WHEN страница отображает заголовок h1 THEN система SHALL использовать стандартный размер IDS (3.6em или класс `.S` для 2.4em) без кастомных переопределений
2. WHEN страница каталога отображает заголовок "AI Каталог для СНГ" THEN система SHALL использовать класс `h1.S` вместо кастомного `font-size: clamp(2em, 5vw, 3.5em)`
3. WHEN HeroSection отображает headline THEN система SHALL использовать стандартный h1 IDS без кастомных clamp-функций
4. WHEN страница категории отображает заголовок THEN система SHALL использовать h1 с размером 2.4em (класс `.S`) вместо кастомного `font-size: 2em`

### Requirement 2: Перегруз Hero-секции

**User Story:** Как пользователь, я хочу видеть минималистичную Hero-секцию, чтобы быстро понять суть сайта без визуального шума.

#### Acceptance Criteria

1. WHEN HeroSection отображается THEN система SHALL использовать только компоненты IDS (Wrapper, TextWidth, Space) без кастомных градиентов
2. WHEN HeroSection содержит CTA-кнопку THEN система SHALL использовать стандартную стилизацию ссылок IDS вместо кастомных button-стилей
3. WHEN HeroSection отображает описание THEN система SHALL использовать класс `.loud` для увеличенного текста вместо кастомных clamp-функций
4. WHEN Hero-секция центрирует контент THEN система SHALL использовать TextWidth без `text-align: center` на всём блоке

### Requirement 3: Абсурдный Sticky Header

**User Story:** Как пользователь, я хочу видеть ненавязчивую навигацию, чтобы она не отвлекала от контента.

#### Acceptance Criteria

1. WHEN GlobalHeader отображается THEN система SHALL использовать стандартный header IDS без `position: sticky`
2. WHEN GlobalHeader имеет фон THEN система SHALL использовать `--ids__background-RGB` вместо `--ids__surface-RGB`
3. WHEN GlobalHeader содержит логотип THEN система SHALL использовать стандартный размер текста без кастомного `font-size: 1.3em`
4. WHEN мобильное меню открывается THEN система SHALL использовать компонент Aside или стандартную навигацию IDS

### Requirement 4: Неконсистентные карточки

**User Story:** Как пользователь, я хочу видеть единообразные карточки инструментов и категорий, чтобы легко сканировать контент.

#### Acceptance Criteria

1. WHEN ToolCard отображает изображение THEN система SHALL использовать компонент Rounded с радиусом `var(--ids__radius)` вместо кастомного `radius="1em"`
2. WHEN CategoryCard отображается THEN система SHALL использовать стандартный aside IDS вместо кастомных стилей с flex
3. WHEN карточка содержит badge/tag THEN система SHALL использовать стандартный code-стиль IDS или mark вместо кастомных badge-стилей
4. WHEN карточка имеет hover-эффект THEN система SHALL использовать только изменение цвета ссылки без transform-анимаций

### Requirement 5: Избыточные кастомные стили

**User Story:** Как разработчик, я хочу использовать только стили IDS, чтобы поддерживать консистентность и уменьшить CSS-код.

#### Acceptance Criteria

1. WHEN компонент использует отступы THEN система SHALL использовать формулу `calc(var(--ids__density) * Nem)` вместо фиксированных значений
2. WHEN компонент использует border-radius THEN система SHALL использовать `var(--ids__radius)` вместо кастомных значений
3. WHEN компонент использует цвета THEN система SHALL использовать только переменные IDS (`--ids__*-RGB`) без хардкода
4. WHEN компонент использует font-size THEN система SHALL использовать em-единицы относительно базового размера IDS

### Requirement 6: Перегруженная страница инструмента

**User Story:** Как пользователь, я хочу видеть чистую страницу инструмента с фокусом на контенте, чтобы быстро получить нужную информацию.

#### Acceptance Criteria

1. WHEN tool page отображает header THEN система SHALL использовать простую вертикальную компоновку вместо grid с двумя колонками
2. WHEN tool page отображает meta-информацию THEN система SHALL группировать badges в один блок aside вместо множества span-элементов
3. WHEN tool page отображает CTA-кнопки THEN система SHALL использовать стандартные ссылки IDS с классом `.ids__promo-link`
4. WHEN tool page отображает FAQ THEN система SHALL использовать компонент details/summary вместо кастомного dl-списка

### Requirement 7: Неконсистентная сетка

**User Story:** Как пользователь, я хочу видеть предсказуемую раскладку контента, чтобы легко находить информацию.

#### Acceptance Criteria

1. WHEN страница использует сетку THEN система SHALL использовать компонент Sequence с размерами S/M/L/XL вместо CSS Grid
2. WHEN страница категории отображает sidebar THEN система SHALL использовать вертикальную компоновку без двухколоночного grid
3. WHEN ToolGrid отображает карточки THEN система SHALL использовать Sequence с SequenceItem вместо кастомного grid
4. WHEN контент требует ограничения ширины THEN система SHALL использовать TextWidth вместо кастомных max-width

### Requirement 8: Избыточные визуальные эффекты

**User Story:** Как пользователь, я хочу видеть спокойный интерфейс без отвлекающих анимаций, чтобы сосредоточиться на контенте.

#### Acceptance Criteria

1. WHEN элемент имеет hover-состояние THEN система SHALL использовать только изменение цвета без transform/scale эффектов
2. WHEN карточка отображает изображение THEN система SHALL отображать статичное изображение без hover-zoom эффекта
3. WHEN страница использует градиенты THEN система SHALL использовать плоские цвета IDS вместо linear-gradient
4. WHEN кнопка имеет hover THEN система SHALL изменять только цвет без translateY эффекта

### Requirement 9: Неконсистентный Footer

**User Story:** Как пользователь, я хочу видеть простой footer с необходимой информацией, чтобы быстро найти нужные ссылки.

#### Acceptance Criteria

1. WHEN GlobalFooter отображается THEN система SHALL использовать простую вертикальную компоновку вместо CSS Grid
2. WHEN footer содержит списки ссылок THEN система SHALL использовать стандартные ul/li IDS без кастомных стилей
3. WHEN footer отображает copyright THEN система SHALL использовать стандартный параграф без центрирования
4. WHEN footer имеет секции THEN система SHALL использовать компонент Space для разделения вместо border-top

### Requirement 10: Перегруженный Blog

**User Story:** Как пользователь, я хочу видеть простой список статей, чтобы быстро найти интересный контент.

#### Acceptance Criteria

1. WHEN BlogCard отображается THEN система SHALL использовать простую вертикальную компоновку без aspect-ratio на изображении
2. WHEN blog post отображает header THEN система SHALL использовать левое выравнивание вместо центрирования
3. WHEN blog post отображает meta THEN система SHALL использовать простой текст без flex-центрирования
4. WHEN blog list отображается THEN система SHALL использовать Sequence с размером XL для вертикального списка

### Requirement 11: Абсурдные размеры шрифтов

**User Story:** Как пользователь, я хочу видеть комфортную типографику, чтобы легко читать контент.

#### Acceptance Criteria

1. WHEN текст отображается THEN система SHALL использовать базовый размер IDS без кастомных font-size
2. WHEN заголовок использует clamp() THEN система SHALL заменить на стандартные размеры IDS с медиа-запросами
3. WHEN описание использует класс .loud THEN система SHALL использовать стандартный размер 2em без модификаций
4. WHEN мелкий текст отображается THEN система SHALL использовать размер не менее 0.85em для читаемости

### Requirement 12: Неиспользуемые компоненты IDS

**User Story:** Как разработчик, я хочу использовать все доступные компоненты IDS, чтобы обеспечить консистентность.

#### Acceptance Criteria

1. WHEN страница содержит важную информацию THEN система SHALL использовать компонент Aside для выделения
2. WHEN страница содержит галерею THEN система SHALL использовать компоненты Gallery/InlineGallery вместо кастомных решений
3. WHEN страница содержит сноски THEN система SHALL использовать компоненты Footnote/FootnoteLink
4. WHEN элементы появляются при скролле THEN система SHALL использовать компонент Sleepy для анимации
