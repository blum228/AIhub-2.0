# Requirements Document

## Introduction

Редизайн страницы сервиса (tool/[slug].astro) для создания профессиональной, SEO-оптимизированной и удобной страницы, которая помогает пользователю принять решение о выборе сервиса. Текущая страница функциональна, но не использует весь потенциал IDS Design System и не оптимизирована для конверсии.

Цели редизайна:
- Улучшить визуальную иерархию и читаемость
- Оптимизировать для SEO (структурированные данные, семантика)
- Повысить конверсию (CTA, социальное доказательство)
- Использовать IDS Design System на 100%

## Glossary

- **IDS Design System**: Дизайн-система проекта с семантическими токенами цветов, компонентами и правилами оформления
- **Hero-секция**: Верхняя часть страницы с основной информацией о сервисе
- **CTA (Call-to-Action)**: Кнопка призыва к действию (переход на сервис)
- **Sticky Sidebar**: Боковая панель, которая остаётся видимой при скролле
- **Schema.org**: Стандарт структурированных данных для SEO
- **Breadcrumbs**: Навигационная цепочка (Главная → Каталог → Категория → Сервис)
- **Access Info**: Блок с информацией о доступности из России (VPN, карты, язык)
- **FAQ Schema**: Структурированные данные для вопросов-ответов (отображаются в Google)

## Requirements

### Requirement 1: Улучшение Hero-секции для первого впечатления

**User Story:** Как посетитель страницы сервиса, я хочу сразу понять что это за сервис и подходит ли он мне, чтобы решить стоит ли читать дальше.

#### Acceptance Criteria

1. WHEN пользователь открывает страницу сервиса THEN система SHALL отображать hero-секцию с названием, кратким описанием, рейтингом и главным CTA
2. WHEN hero-секция отображается THEN система SHALL показывать изображение сервиса с правильным соотношением сторон и скруглёнными углами по IDS
3. WHEN сервис имеет рейтинг THEN система SHALL отображать рейтинг в виде звёзд с числовым значением
4. WHEN пользователь видит hero THEN система SHALL показывать ключевые бейджи доступности (VPN, карты РФ, цена) в компактном виде

### Requirement 2: Оптимизация блока доступности для пользователей из России

**User Story:** Как пользователь из России, я хочу сразу видеть информацию о доступности сервиса, чтобы понять смогу ли я им пользоваться.

#### Acceptance Criteria

1. WHEN страница сервиса загружается THEN система SHALL отображать блок Access Info с информацией о VPN, картах РФ, языке и способах оплаты
2. WHEN сервис требует VPN THEN система SHALL показывать предупреждение с цветовой индикацией (warning) и краткой инструкцией
3. WHEN сервис не принимает карты РФ THEN система SHALL показывать альтернативные способы оплаты (крипто, виртуальные карты)
4. WHEN блок Access Info отображается THEN система SHALL использовать семантические цвета IDS (success/warning/info)

### Requirement 3: Улучшение структуры контента для читаемости

**User Story:** Как пользователь, я хочу легко находить нужную информацию на странице, чтобы быстро принять решение.

#### Acceptance Criteria

1. WHEN пользователь просматривает страницу THEN система SHALL отображать контент в двухколоночном layout (основной контент + sidebar)
2. WHEN контент отображается THEN система SHALL использовать типографику IDS с правильными отступами и размерами
3. WHEN страница содержит таблицы THEN система SHALL стилизовать их по IDS с чёткими границами и читаемым текстом
4. WHEN страница содержит списки THEN система SHALL использовать стилизацию списков IDS

### Requirement 4: Создание эффективного Sticky Sidebar

**User Story:** Как пользователь, я хочу иметь быстрый доступ к CTA и ключевой информации при скролле, чтобы перейти на сервис в любой момент.

#### Acceptance Criteria

1. WHEN пользователь скроллит страницу THEN система SHALL показывать sticky sidebar с CTA-кнопкой
2. WHEN sidebar отображается THEN система SHALL показывать краткую информацию о цене и способах оплаты
3. WHEN сервис имеет ограничения (VPN, оплата) THEN sidebar SHALL показывать краткие подсказки по обходу
4. WHEN экран мобильный THEN система SHALL скрывать sticky sidebar и показывать фиксированную CTA-кнопку внизу

### Requirement 5: Оптимизация FAQ-секции для SEO

**User Story:** Как владелец сайта, я хочу чтобы FAQ отображался в результатах Google, чтобы привлекать больше трафика.

#### Acceptance Criteria

1. WHEN страница содержит FAQ THEN система SHALL генерировать FAQPage Schema.org разметку
2. WHEN FAQ отображается THEN система SHALL использовать аккордеон с плавной анимацией
3. WHEN FAQ-элемент раскрыт THEN система SHALL показывать ответ с правильной типографикой IDS
4. WHEN FAQ пуст THEN система SHALL не отображать секцию FAQ

### Requirement 6: Улучшение секции похожих сервисов

**User Story:** Как пользователь, я хочу видеть альтернативы текущему сервису, чтобы сравнить и выбрать лучший вариант.

#### Acceptance Criteria

1. WHEN страница сервиса отображается THEN система SHALL показывать секцию "Похожие сервисы" с 3-4 релевантными альтернативами
2. WHEN похожие сервисы отображаются THEN система SHALL показывать карточки с ключевой информацией (название, цена, бейджи доступности)
3. WHEN похожие сервисы выбираются THEN система SHALL приоритизировать сервисы из тех же коллекций и с похожими тегами

### Requirement 7: Добавление структурированных данных для SEO

**User Story:** Как владелец сайта, я хочу чтобы страницы сервисов хорошо индексировались и отображались в поиске, чтобы привлекать органический трафик.

#### Acceptance Criteria

1. WHEN страница сервиса загружается THEN система SHALL генерировать Product Schema.org с названием, описанием, рейтингом и ценой
2. WHEN страница имеет breadcrumbs THEN система SHALL генерировать BreadcrumbList Schema.org
3. WHEN страница имеет FAQ THEN система SHALL генерировать FAQPage Schema.org
4. WHEN структурированные данные генерируются THEN система SHALL валидировать их корректность

### Requirement 8: Адаптивность для мобильных устройств

**User Story:** Как мобильный пользователь, я хочу удобно просматривать страницу сервиса на телефоне, чтобы принять решение в любом месте.

#### Acceptance Criteria

1. WHEN экран меньше 768px THEN система SHALL переключать layout на одноколоночный
2. WHEN экран мобильный THEN система SHALL показывать фиксированную CTA-кнопку внизу экрана
3. WHEN hero-секция на мобильном THEN система SHALL располагать изображение над текстом
4. WHEN таблицы на мобильном THEN система SHALL обеспечивать горизонтальный скролл или адаптивное отображение
